# JsonLd Component

The skill creates `components/JsonLd.tsx` once per project as a shared utility for rendering JSON-LD schema. Every optimized page imports this component to inject structured data.

This file specifies the component's exact source code, props, and usage rules.

---

## Why a wrapper component

Three reasons:

1. **Type safety.** The component types the `data` prop and prevents accidental passing of non-serializable values.
2. **Consistency.** Every schema script across the site renders identically, with the same `application/ld+json` MIME type and the same JSON.stringify formatting.
3. **Server-rendered.** As a Server Component (App Router) or simple component (Pages Router), it renders once at build/request time and never adds client-side JavaScript.

---

## The file the skill creates

**Path:** `components/JsonLd.tsx`

If the project's components directory is elsewhere (e.g., `src/components/`, `app/_components/`), the skill detects the convention from existing files and writes there. Default is `components/` at project root.

```tsx
type SchemaValue =
  | string
  | number
  | boolean
  | null
  | SchemaValue[]
  | { [key: string]: SchemaValue };

type JsonLdProps = {
  data: SchemaValue;
};

/**
 * Renders a JSON-LD <script> tag for structured data.
 *
 * Generated by the pracpros-dental-aeo skill. Do not edit manually unless
 * the skill is no longer in use on this project.
 *
 * Usage:
 *   const schema = { "@context": "https://schema.org", "@graph": [...] };
 *   return <JsonLd data={schema} />;
 *
 * Renders as:
 *   <script type="application/ld+json">{...}</script>
 */
export function JsonLd({ data }: JsonLdProps) {
  return (
    <script
      type="application/ld+json"
      // Schema.org JSON-LD must be inline; React text-node escaping would break valid JSON.
      // dangerouslySetInnerHTML is the standard pattern for this. Data is constructed
      // server-side from typed values, never from user input.
      dangerouslySetInnerHTML={{ __html: JSON.stringify(data) }}
    />
  );
}
```

## TypeScript-strict project handling

If the project uses `strict: true` and `noUncheckedIndexedAccess: true` in `tsconfig.json`, the recursive `SchemaValue` type above compiles cleanly without modification. No need for `unknown` casts or `any`.

If the project does not use TypeScript (the file is `.jsx` instead of `.tsx`), the skill writes the equivalent JavaScript:

```jsx
/**
 * Renders a JSON-LD <script> tag for structured data.
 *
 * Generated by the pracpros-dental-aeo skill. Do not edit manually unless
 * the skill is no longer in use on this project.
 */
export function JsonLd({ data }) {
  return (
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(data) }}
    />
  );
}
```

---

## Why `dangerouslySetInnerHTML`

The `dangerouslySetInnerHTML` API name sounds alarming. It is the documented React pattern for inline JSON-LD scripts.

The alternative — placing JSON inside `<script>{JSON.stringify(data)}</script>` as a child node — fails because React HTML-escapes string children. JSON-LD with HTML-escaped quotes is invalid JSON and will be silently dropped by Google's parsers.

`dangerouslySetInnerHTML` is safe in this specific use case because:

- The `data` argument is constructed at skill-execution time from CLIENT_BRIEF values, not from user input.
- JSON.stringify on typed data cannot inject script tags (it escapes `<`, `>`, and `&` as Unicode escapes inside string values).
- The script type is `application/ld+json`, which is not executed as JavaScript by browsers.

The skill never accepts user input directly into the schema constant. The data is always derived from CLIENT_BRIEF.md, which is committed to the project repo by the developer.

## Usage in optimized pages

The skill writes `import { JsonLd } from "@/components/JsonLd";` at the top of every optimized page. Resolution rules:

1. If `tsconfig.json` declares `paths` with `"@/*": ["./*"]` (Next.js default since v13), use `@/components/JsonLd`.
2. If no path alias is configured, use a relative import: `../../components/JsonLd` for `app/services/dental-implants/page.tsx`, computed per-file.

The component is rendered as a JSX element near the top of the page's return value:

```tsx
return (
  <>
    <JsonLd data={schema} />
    <article>{/* page content */}</article>
  </>
);
```

For Pages Router, same usage:

```tsx
return (
  <>
    <Head>{/* meta tags */}</Head>
    <JsonLd data={schema} />
    <article>{/* page content */}</article>
  </>
);
```

## Multiple schemas per page

The skill prefers one `<JsonLd>` component per page with all schemas combined into a single `@graph` array:

```tsx
const schema = {
  "@context": "https://schema.org",
  "@graph": [
    { "@type": "Service", "@id": "...", /* ... */ },
    { "@type": "FAQPage", "@id": "...", /* ... */ },
    { "@type": "BreadcrumbList", "@id": "...", /* ... */ },
  ],
};

return <JsonLd data={schema} />;
```

Reasons to prefer one `<JsonLd>` over multiple:

- One `@graph` lets schemas reference each other via `@id` cleanly.
- Cleaner page source.
- Matches Google's recommended pattern for combined schemas.

The component supports multiple `<JsonLd>` elements per page if the developer needs them, but the skill always emits one.

## When the skill updates an existing JsonLd component

If `components/JsonLd.tsx` already exists in the project, the skill checks:

1. **Does it accept a `data` prop?** If yes, leave it alone.
2. **Does it use `dangerouslySetInnerHTML` (or similar inline rendering)?** If yes, leave it alone.
3. **Does it work for both server and client components?** If yes, leave it alone.
4. **Otherwise**, the skill writes its version with a comment at the top noting the previous file is in `components/JsonLd.legacy.tsx` (the skill renames, never deletes).

If a different filename in the project already provides equivalent functionality (e.g., `components/StructuredData.tsx`, `components/Schema.tsx`), the skill detects it and uses the existing component:

```
[COMPONENT REUSE] Found existing components/StructuredData.tsx with compatible API.
                  Skill is using StructuredData instead of creating JsonLd.
                  All optimized pages import { StructuredData } from "@/components/StructuredData".
```

## What the JsonLd component does NOT do

- It does not validate the schema. Validation happens earlier in the skill workflow (Step 4h, before writing).
- It does not deduplicate `@id` values. The skill ensures uniqueness during schema generation.
- It does not minify or pretty-print the JSON. `JSON.stringify(data)` produces minified output. For developer debugging, the developer can use `JSON.stringify(data, null, 2)` locally.
- It does not handle `nonce` for CSP-strict environments. If the project has a strict Content Security Policy, the developer adds `nonce` separately. The skill flags CSP needs in the run report when it detects a `Content-Security-Policy` header configured in `next.config.js` or middleware.
